你是一个资深全栈工程师。请帮我实现一个“Bilibili 异地一起看”的最小可行版本（MVP）。

## 产品目标

做一个能让我和对象异地一起看 Bilibili 视频的最小实现方案，要求是：

- 不要尝试抓取、下载、转存、转发、破解、重放 Bilibili 视频源
- 不要自己实现独立视频播放器
- 只在用户各自正常打开的 Bilibili 视频页面内工作
- 通过用户脚本监听和控制当前页面里的播放器状态，实现两人同步观看
- 两个人分别在自己的浏览器里打开同一个 Bilibili 视频页，加入同一个房间后，其中一方操作播放/暂停/拖动进度/切分P，另一方自动同步
- 仅做 MVP，优先保证结构清晰、代码能跑、方便本地调试
- 不做语音通话、不做聊天、不做弹幕同步、不做移动端适配
- 不做复杂数据库，不做用户账号系统

## 技术方案

请直接生成一个可运行项目，分成两个目录：

1. `server/`
   - Node.js + TypeScript
   - 使用 `ws` 实现一个极简 WebSocket 房间服务
   - 不使用数据库，房间状态放内存即可
   - 提供房间加入、离开、广播同步事件、房主机制、心跳检测
   - 默认端口 8787
   - 提供 README，说明如何启动

2. `userscript/`
   - Tampermonkey 用户脚本
   - 只匹配 Bilibili 视频页，例如 `https://www.bilibili.com/video/*`
   - 在页面右上角或右下角插入一个简洁的控制面板
   - 支持输入房间号、加入房间、离开房间、显示连接状态、设置自己是否为房主
   - 连接 WebSocket 服务后，监听本地播放器事件并广播给房间内其他人
   - 接收到远端事件后，控制本地播放器同步执行

## 必须实现的功能

### 1. 房间与角色
- 用户输入一个 roomId 即可加入房间
- 房间中只允许一个“主控”
- 默认第一个加入的人是主控
- 非主控用户默认只跟随，不主动广播播放控制事件
- 提供“申请成为主控”或“切换主控”的简单按钮，但实现尽量简单
- 服务端维护房间当前主控连接 ID
- 主控断开后，房间内下一个在线成员自动成为主控

### 2. 同步事件
只实现这几类事件：

- `join`
- `leave`
- `state`
- `play`
- `pause`
- `seek`
- `changePart`
- `ping`
- `pong`
- `masterChanged`
- `error`

事件负载至少包含：
- `roomId`
- `senderId`
- `videoKey`
- `timestamp`
- 以及具体事件所需字段

### 3. videoKey 设计
请设计一个 `videoKey`，用于识别当前 Bilibili 视频上下文，至少包含：
- `bvid`（如果能从 URL 拿到）
- `p`（分P序号，如果存在）
- 当前 URL

如果拿不到完整信息，也要优雅降级，至少保证 URL 可用。

### 4. 本地播放器接入
不要假设 Bilibili 页面一定暴露稳定官方 API。
实现时请采用“多策略探测 + 可降级”的方式：

优先顺序建议：
1. 尝试找到页面上的 `<video>` 元素
2. 如果有多个 video，选择可见且尺寸最大的那个
3. 用 HTMLMediaElement 的标准能力进行控制：
   - `play()`
   - `pause()`
   - `currentTime`
   - `paused`
4. 分P切换优先通过 URL 或页面中的分P链接识别和跳转
5. 不依赖私有、脆弱、写死的全局播放器对象名
6. 如果当前页面无法找到可控播放器，要在 UI 中明确提示“未检测到播放器”

### 5. 事件监听与同步策略
本地监听：
- `play`
- `pause`
- `seeking`
- `seeked`
- `loadedmetadata`
- URL 变化（用于识别切分P或切视频）

远端同步：
- 收到 `play` 时：
  - 若本地与目标时间差 > 1 秒，先校准 `currentTime`
  - 再尝试播放
- 收到 `pause` 时：
  - 如有明显时间差，先校准时间
  - 再暂停
- 收到 `seek` 时：
  - 将本地 `currentTime` 设置到目标时间
  - 若当前应播放则尽量保持播放态
- 收到 `changePart` 时：
  - 如果 URL 不同，则跳转到对方对应的 URL
  - 跳转后等待页面稳定，再继续同步

### 6. 防抖与避免回声
必须解决“同步回声”和“双方互相抢控制”的问题：

- 引入 `isApplyingRemoteEvent` 标记
- 当脚本正在执行远端同步时，忽略本地事件上报
- 对 `seek` 做防抖，拖动过程中不要疯狂广播
- 只有拖动完成后或间隔一定时间才广播
- 对小误差做容忍：例如时间差小于 0.8 秒不修正
- 非主控端默认不广播播放控制事件
- 本地自动同步后，短时间内不要再次将同一动作反向广播出去

### 7. 自动播放失败处理
浏览器可能阻止自动播放。
请实现：
- 远端发来 `play` 时，如果 `video.play()` 被拒绝，捕获异常
- 在用户脚本 UI 中显示提示：“浏览器阻止了自动播放，请手动点击一次播放以授予权限”
- 不要让脚本报错崩溃

### 8. UI 需求
用户脚本面板至少包括：
- 房间号输入框
- 连接/断开按钮
- 当前连接状态
- 当前是否为主控
- 当前检测到的 videoKey
- 最近一条同步日志
- 一个“重新检测播放器”按钮

UI 要求：
- 简洁
- 不遮挡主要播放区
- 用原生 DOM + 少量内联样式即可，不引入任何前端框架

### 9. WebSocket 服务端要求
服务端需要：
- 维护连接 ID
- 维护房间成员列表
- 维护房间主控 ID
- 广播主控的同步事件给房间内其他成员
- 支持心跳检测，清理失效连接
- 连接断开后自动从房间移除
- 主控断开后自动选出新主控并广播 `masterChanged`
